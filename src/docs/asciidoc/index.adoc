== Overview

`{project-name}` is an async Java client for Meshtastic radios that provides:

- transport and reconnect management
- startup synchronization
- strongly-typed event dispatching
- node database + snapshot status model
- admin/config/channel APIs through `AdminService`
- utility request APIs (fire-and-forget and payload-await)
- OTA orchestration framework with pluggable upload strategies
- BLE transport SPI for platform-specific BLE backends

Javadocs are generated at link:{javadocs-url}[API Javadocs].

== Current Scope And Gaps

This project is production-focused for serial/TCP Meshtastic clients.

- Production-ready now:
  - Serial and TCP transports
  - Event-driven client model
  - Admin/config/channel refresh + write APIs
  - Request lifecycle + startup lifecycle signaling

- Pending major items:
  - bundled production BLE backend (BLE SPI exists)
  - full production OTA flashing integration (framework exists)

[NOTE]
====
For first public release positioning, this library is best described as serial/TCP-first, with BLE and OTA remaining as the largest feature-completeness items.
====

[IMPORTANT]
====
BLE and OTA should currently be understood as supported extension points rather than fully turnkey, end-to-end
features inside this repository. The public APIs are present, but they still require concrete backend/uploader
implementations validated for the target platform and hardware.
====

== API Catalog (By Class)

Use this quick map when building app-level abstractions:

- `MeshtasticClient`
  Handles transport lifecycle, startup sync, text send helpers, utility node-info/position/telemetry requests, and event fan-out.
- `MeshtasticEventListener`
  Unified callback surface for startup state, request lifecycle, admin model updates, and mesh data events.
- `AdminService`
  Typed refresh/read/write operations for owner, metadata, channels, config sections, and module configs.
- `ProtocolConstraints`
  Pre-validation helpers for channel index/name/PSK and non-throwing validation issues for form UIs.
- `NodeDatabase` + `MeshNode`
  Snapshot store and node freshness/status modeling (`LIVE/IDLE/CACHED/OFFLINE`).
- `OtaService`
  OTA orchestration that composes admin OTA-mode request with pluggable upload strategies.
- `SerialTransport` / `TcpTransport` / `BleTransport`
  Physical transport adapters with framing, reconnect behavior, and pacing controls.

== Quick Start

[source,java]
----
NodeDatabase db = new InMemoryNodeDatabase();
MeshtasticClient client = new MeshtasticClient(db);

SerialConfig cfg = SerialConfig.builder()
        .portName("/dev/cu.usbmodem80B54ED11F101")
        // Optional transport pacing between framed writes (default: 200ms)
        .outboundPacingDelayMs(200)
        .build();

// Optional advanced request-timing tuning (defaults shown)
client.setRequestLockReleaseCooldownMs(200);               // default: 200ms
client.setRequestCorrelationTimeout(Duration.ofSeconds(30)); // default: 30s

client.connect(new SerialTransport(cfg));
----

[NOTE]
====
Keep defaults unless you have measured evidence your radio/firmware tolerates tighter pacing.
====

[TIP]
====
Order matters for some settings:

- Transport config such as `SerialConfig.outboundPacingDelayMs`, `TcpConfig.outboundPacingDelayMs`,
  BLE reconnect/backoff settings, and other transport-builder values are startup-time settings. Set them before
  `connect(...)`. To change them later, rebuild/reconnect the transport.
- `MeshtasticClient` request timing knobs such as `setRequestLockReleaseCooldownMs(...)` and
  `setRequestCorrelationTimeout(...)` are runtime-tunable and can be changed before or after `connect(...)`.
- `AdminService.setVerificationPolicy(...)` is also runtime-tunable and affects future verify-applied admin writes.
====

== Startup Flow (Recommended)

Use startup state events to drive UI readiness and progress indicators.

[source,java]
----
client.addEventListener(new MeshtasticEventListener() {
    @Override
    public void onStartupStateChanged(StartupState previousState, StartupState newState) {
        switch (newState) {
            case SYNC_LOCAL_CONFIG -> statusLabel.setText("Reading local radio...");
            case SYNC_MESH_CONFIG -> statusLabel.setText("Reading mesh state...");
            case READY -> statusLabel.setText("Ready");
            case DISCONNECTED -> statusLabel.setText("Disconnected");
        }
    }
});

client.connect(new SerialTransport(cfg));
----

For tab gating, use `client.isReady()` and keep settings/message actions disabled until `READY`.

== UI Client Workflow

This workflow is recommended for desktop/mobile UI applications.

1. Create `NodeDatabase` and `MeshtasticClient`.
2. Register one event listener implementation that forwards to your UI state/store.
3. Connect transport.
4. Gate settings/actions until `StartupState.READY`.
5. Render settings pages from snapshots first, then trigger section-specific refresh.

[source,java]
----
NodeDatabase db = new InMemoryNodeDatabase();
MeshtasticClient client = new MeshtasticClient(db);
AdminService admin = client.getAdminService();

client.addEventListener(new MeshtasticEventListener() {
    @Override
    public void onStartupStateChanged(StartupState previous, StartupState next) {
        switch (next) {
            case SYNC_LOCAL_CONFIG -> uiStore.setStatus("Reading local radio...");
            case SYNC_MESH_CONFIG -> uiStore.setStatus("Reading mesh state...");
            case READY -> uiStore.setStatus("Ready");
            case DISCONNECTED -> uiStore.setStatus("Disconnected");
        }
    }

    @Override
    public void onNodeDiscovery(NodeDiscoveryEvent event) {
        uiStore.upsertNode(event.getNode());
    }

    @Override
    public void onTelemetryUpdate(TelemetryUpdateEvent event) {
        uiStore.updateTelemetry(event.getNodeId(), event.getTelemetry());
    }

    @Override
    public void onPositionUpdate(PositionUpdateEvent event) {
        uiStore.updatePosition(event.getNodeId(), event.getPosition());
    }

    @Override
    public void onAdminModelUpdate(AdminModelUpdateEvent event) {
        // Keep settings pages in sync while app is connected.
        uiStore.markAdminSectionDirty(event.getSection(), event.getChannelIndex());
    }

    @Override
    public void onRequestLifecycleUpdate(RequestLifecycleEvent event) {
        uiStore.logRequest(
                event.getStage() + " id=" + event.getRequestId()
                        + " port=" + event.getPort()
                        + " msg=" + event.getMessage()
        );
    }
});

client.connect(new SerialTransport(cfg));
----

== Headless Client Workflow

This workflow is recommended for bots, services, and integrations without a GUI.

1. Connect client with the target transport.
2. Await `READY` before operational tasks.
3. Use listeners for streaming behavior (telemetry, text, node updates).
4. Use scheduled jobs for periodic refresh/write logic.
5. Shutdown cleanly with `client.shutdown()`.

[source,java]
----
NodeDatabase db = new InMemoryNodeDatabase();
MeshtasticClient client = new MeshtasticClient(db);
AdminService admin = client.getAdminService();

CountDownLatch readyLatch = new CountDownLatch(1);

client.addEventListener(new MeshtasticEventListener() {
    @Override
    public void onStartupStateChanged(StartupState previous, StartupState next) {
        if (next == StartupState.READY) {
            readyLatch.countDown();
        }
    }

    @Override
    public void onTextMessage(ChatMessageEvent event) {
        botEngine.handleInboundMessage(event);
    }
});

client.connect(new SerialTransport(cfg));
readyLatch.await(45, TimeUnit.SECONDS);

// Periodic task: refresh security + mqtt every 10 minutes.
ScheduledExecutorService jobs = Executors.newSingleThreadScheduledExecutor();
jobs.scheduleAtFixedRate(() ->
        admin.refreshSecurityConfig()
             .thenCompose(cfgSec -> admin.refreshMqttConfig())
             .exceptionally(ex -> {
                 log.warn("Periodic refresh failed: {}", ex.getMessage());
                 return null;
             }),
        0, 10, TimeUnit.MINUTES
);
----

Hardware-in-the-loop tests are intentionally opt-in and isolated from default unit tests.

- Maven profile: `hardware-it`
- Test class: `RealRadioAdminIT`
- Required environment variable: `MESHTASTIC_TEST_PORT`
- Optional environment variables:
  - `MESHTASTIC_TEST_TIMEOUT_SEC` (default `45`)
  - `MESHTASTIC_TEST_MUTABLE_CHANNEL_INDEX` (default `2`)
  - `MESHTASTIC_TEST_ENABLE_OWNER_WRITE` (default `false`)
  - `MESHTASTIC_TEST_ENABLE_SECURITY_WRITE` (default `false`)
  - `MESHTASTIC_TEST_ENABLE_MQTT_WRITE` (default `false`)
  - `MESHTASTIC_TEST_ENABLE_REBOOT_TEST` (default `false`)

[source,bash]
----
MESHTASTIC_TEST_PORT=/dev/cu.usbmodem80B54ED11F101 \
mvn -Phardware-it verify
----

[NOTE]

The reversible write test restores original channel state after verification and should target an active non-primary channel slot.
====

Coverage in current hardware integration tests:

- startup readiness (`READY`)
- metadata/owner/likely-channel read path
- core config read matrix
- module config read matrix (capability tolerant)
- request burst lifecycle terminal events
- reversible channel write/readback/restore
- optional reversible owner write/readback/restore (enabled with `MESHTASTIC_TEST_ENABLE_OWNER_WRITE=true`)
- optional reversible security config write/readback/restore (enabled with `MESHTASTIC_TEST_ENABLE_SECURITY_WRITE=true`)

== Core API Patterns

=== Transport and Request Timing Controls

The client intentionally serializes in-flight requests for radio safety. Timing controls let you tune throughput
without changing that safety model.

==== Transport Pacing

`SerialConfig` and `TcpConfig` expose:

- `outboundPacingDelayMs` (default `200`)

Usage:

[source,java]
----
SerialConfig serialCfg = SerialConfig.builder()
        .portName("/dev/cu.usbmodem80B54ED11F101")
        .outboundPacingDelayMs(120) // lower for faster radios, 0 disables pacing
        .build();

TcpConfig tcpCfg = TcpConfig.builder()
        .host("192.168.1.50")
        .port(4403)
        .outboundPacingDelayMs(120)
        .build();
----

These are startup-time transport settings. They are read from the transport config during transport construction/connect.
If you want to change them later, create a new transport/config and reconnect.

==== Request Correlation Controls

`MeshtasticClient` exposes:

- `setRequestLockReleaseCooldownMs(long)` (default `200ms`)
- `setRequestCorrelationTimeout(Duration)` (default `30s`)

Usage:

[source,java]
----
client.setRequestLockReleaseCooldownMs(120);
client.setRequestCorrelationTimeout(Duration.ofSeconds(20));
----

These settings are runtime-tunable. You can change them before or after `connect(...)`, and future requests will use
the updated values.

==== Admin Verification Policy

`AdminService` exposes:

- `setVerificationPolicy(AdminVerificationPolicy)`

Usage:

[source,java]
----
client.getAdminService().setVerificationPolicy(
        AdminVerificationPolicy.builder()
                .maxAttempts(3)
                .initialRetryDelay(Duration.ofMillis(400))
                .retryBackoffMultiplier(1.8)
                .maxRetryDelay(Duration.ofSeconds(2))
                .build()
);
----

This setting is also runtime-tunable and affects future `verifyApplied=true` admin writes.

Example TRACE output when timing diagnostics are enabled:

[source,text]
----

[TX] Request 1856599835 completed in 6450ms (port=ADMIN_APP dst=!53ad0f1e success=true)
[UTIL] Await TELEMETRY_APP for !699c5400 completed in 312ms (success=true)
----

=== Fire-and-forget Request Correlation

Use request methods when you need transport/routing acceptance only:

[source,java]
----
client.requestNodeInfo(nodeId);
client.requestPosition(nodeId);
client.requestTelemetry(nodeId);
----

=== Payload-Await Requests

Use payload-await methods when the caller must wait for live payload-driven node updates:

[source,java]
----
client.requestNodeInfoAwaitPayload(nodeId);
client.requestPositionAwaitPayload(nodeId);
client.requestTelemetryAwaitPayload(nodeId);
----

=== Payload-Await with Snapshot Fallback

When firmware/device timing is inconsistent, use `...AwaitPayloadOrSnapshot(...)` helpers.
These complete with latest node snapshot when a live payload is not observed before timeout.

[source,java]
----
client.requestNodeInfoAwaitPayloadOrSnapshot(nodeId, Duration.ofSeconds(15));
client.requestPositionAwaitPayloadOrSnapshot(nodeId, Duration.ofSeconds(15));
client.requestTelemetryAwaitPayloadOrSnapshot(nodeId, Duration.ofSeconds(15));
----

=== Messaging APIs

Use convenience methods on `MeshtasticClient` for common text workflows:

[source,java]
----
// 1) Typical direct message to a single node on the primary channel.
client.sendDirectText(targetNodeId, "hello from java");

// 2) Broadcast to the public/default chat on the primary channel.
client.sendChannelText(MeshConstants.PRIMARY_CHANNEL_INDEX, "hello mesh");

// 3) Broadcast to a specific shared channel such as channel 1 ("Test").
client.sendChannelText(1, "hello test channel");

// 4) Advanced: direct message to one node while using a non-primary
// shared channel context for encryption/key selection.
client.sendDirectText(targetNodeId, 1, "hello on channel 1");
----

Both methods automatically apply chunking for long payloads.

[NOTE]
====
For direct messages, the destination node controls whether the packet is one-to-one, while the channel index
selects the shared channel/key context used for that packet. For channel messages, the destination is broadcast
and the channel index selects which group channel receives the message.
====

==== Which Method Should I Use?

[cols="1,3",options="header"]
|===
| Goal
| Method

| Send a private message to one device using the normal/default path
| `client.sendDirectText(nodeId, "hello")`

| Send a message to everyone on the public/default channel
| `client.sendChannelText(MeshConstants.PRIMARY_CHANNEL_INDEX, "hello everyone")`

| Send a message to everyone on a secondary channel such as "Test"
| `client.sendChannelText(channelIndex, "hello test")`

| Send a private message to one device while using a specific shared channel's key/context
| `client.sendDirectText(nodeId, channelIndex, "hello")`
|===

==== UI Example

This is a common UI pattern:

- DM conversation view: call `sendDirectText(selectedNodeId, message)`
- Public chat view: call `sendChannelText(MeshConstants.PRIMARY_CHANNEL_INDEX, message)`
- Secondary channel chat view (for example "Test"): call `sendChannelText(selectedChannel.index(), message)`

[source,java]
----
if (conversation.isDirect()) {
    client.sendDirectText(conversation.targetNodeId(), messageText);
} else {
    client.sendChannelText(conversation.channelIndex(), messageText);
}
----

Bots often need both broadcast and direct-reply behavior.

=== Admin Service Access

`AdminService` is obtained from the client:

[source,java]
----
AdminService admin = client.getAdminService();
----

`AdminService` is the high-level settings/configuration facade for the library.
It exists so application code does not have to manually build every admin protobuf request,
track every response variant, merge incoming admin snapshots into a coherent radio model, or
re-implement write verification rules.

In practice, `MeshtasticClient` and `AdminService` have different jobs:

- `MeshtasticClient` owns transport lifecycle, startup sync, request correlation, messaging helpers, and event fan-out.
- `AdminService` owns radio settings/domain state: owner, metadata, channels, config sections, module configs, and
  admin-oriented write/readback workflows.

Why this layer exists:

- Meshtastic admin/config data arrives across multiple message types and at multiple times.
- Settings UIs usually need both instant cached reads and explicit refresh-on-open behavior.
- Admin writes often need eventual-consistency handling because some radios apply writes asynchronously.
- It is easier for app developers to work with a stable service-oriented API than with dozens of raw admin message variants.

What `AdminService` gives you:

- a live `RadioModel` snapshot via `getSnapshot()`
- typed snapshot accessors for channels/config/module-config sections
- explicit async refresh methods for page-specific data loads
- typed write helpers for channel/config/module/owner operations
- optional verify-applied semantics with structured `AdminWriteResult`

Mental model:

- use `AdminService` as your settings/read-write facade
- use `MeshtasticEventListener.onAdminModelUpdate(...)` to keep UI state in sync while connected
- use snapshots for instant rendering
- use refresh methods when a page opens or when a user explicitly asks to reload

This split keeps the lower-level client API clean while still giving UI and headless applications
a practical, higher-level surface for Meshtastic settings workflows.

==== Snapshot Model And Refresh Model

`AdminService` is intentionally snapshot-first.

- Snapshot reads are immediate and are ideal for first paint in a UI.
- Refresh methods are explicit and async so callers control when radio traffic is generated.
- Incoming admin/local-state packets continuously update the in-memory `RadioModel`.

Typical pattern:

1. render immediately from snapshot
2. trigger a focused refresh for the active page
3. listen for `onAdminModelUpdate(...)` to keep the view/store current

This avoids both extremes:

- blocking UI on a full radio sweep every time
- forcing application code to manually merge incoming admin packets

==== Who Should Use It?

Use `AdminService` when your application needs:

- settings screens
- channel editors
- owner/metadata display
- config/module configuration
- lifecycle admin actions such as reboot/factory reset/OTA-mode request

If you are building a simple bot or telemetry consumer and do not need radio settings,
you may spend most of your time in `MeshtasticClient` plus event listeners and rarely touch `AdminService`.

== Settings Page Patterns

=== Read-on-open (Snapshot + Refresh)

Render from snapshot immediately, then refresh only the page-specific state:

[source,java]
----
AdminService admin = client.getAdminService();

// Immediate render from cached state
renderChannels(admin.getChannelSnapshots());
renderConfig(admin.getConfigSnapshots());
renderModules(admin.getModuleConfigSnapshots());

// On-demand refresh for a channels/security page
admin.refreshChannelsAndSecurity().thenAccept(model -> {
    renderChannels(admin.getChannelSnapshots());
    renderSecurity(admin.getConfigSnapshot(ConfigType.SECURITY_CONFIG).orElse(null));
    renderMqtt(admin.getModuleConfigSnapshot(ModuleConfigType.MQTT_CONFIG).orElse(null));
});

// Explicit deep refresh (all channel slots 0..7)
admin.refreshAllChannelsAndSecurity().thenAccept(model -> {
    renderChannels(admin.getChannelSnapshots());
});
----

=== Module/Config Refresh by Section

[source,java]
----
admin.refreshCore();            // metadata + owner + core config blocks
admin.refreshCoreConfigs();     // lora/device/display/network
admin.refreshModules();         // all supported module config sections
admin.refreshMqttConfig();      // mqtt only
admin.refreshSecurityConfig();  // security only
----

== Admin Write Semantics

Canonical write entry points:

- `setChannelResult(index, channel, verifyApplied)`
- `setConfigResult(config, verifyApplied)`
- `setModuleConfigResult(moduleConfig, verifyApplied)`
- `setOwnerResult(targetNodeId, longName, shortName, verifyApplied)`

Boolean-oriented variants are also available when structured status is not needed:

- `setChannel(index, channel, verifyApplied)`
- `setConfig(config, verifyApplied)`
- `setModuleConfig(moduleConfig, verifyApplied)`
- `setOwner(targetNodeId, longName, shortName, verifyApplied)`

Accepted-only and verified semantics are expressed through the explicit `verifyApplied` boolean
or structured `AdminWriteResult` status.

=== Eventual-Consistency Verification Policy

Some firmware/radio combinations apply writes asynchronously. To reduce false negatives,
`AdminService` supports configurable read-back retries when `verifyApplied=true`.

Defaults:

- `maxAttempts = 1` (immediate single verification attempt; legacy behavior)
- `initialRetryDelay = 300ms`
- `retryBackoffMultiplier = 2.0`
- `maxRetryDelay = 2s`

Configuration example:

[source,java]
----
admin.setVerificationPolicy(
        AdminVerificationPolicy.builder()
                .maxAttempts(3)
                .initialRetryDelay(Duration.ofMillis(400))
                .retryBackoffMultiplier(1.8)
                .maxRetryDelay(Duration.ofSeconds(2))
                .build()
);
----

Operational guidance:

- keep `maxAttempts=1` for fastest fail-fast behavior
- use `2-3` attempts for unstable radios/firmware where writes may appear late
- avoid high retry counts to prevent blocking settings workflows

=== Structured Write Results (Recommended for UI)

For consistent error/status handling, use result-returning write methods:

[source,java]
----
CompletableFuture<AdminWriteResult> resultFuture =
        admin.setConfigResult(configPayload, true);

resultFuture.thenAccept(result -> {
    switch (result.status()) {
        case VERIFIED_APPLIED -> showSuccess("Applied and verified.");
        case ACCEPTED -> showSuccess("Accepted by radio.");
        case REJECTED -> showError("Rejected: " + result.message());
        case TIMEOUT -> showError("Timed out: " + result.message());
        case VERIFICATION_FAILED -> showWarning("Accepted but read-back mismatch.");
        case FAILED -> showError("Write failed: " + result.message());
    }
});
----

Available result statuses:

- `ACCEPTED`
- `VERIFIED_APPLIED`
- `REJECTED`
- `TIMEOUT`
- `VERIFICATION_FAILED`
- `FAILED`

  Admin --> UI: true/false
end

@enduml
----

== Settings Cookbook

This section shows practical UI workflows by page. The pattern is consistent:

1. Render immediately from snapshots.
2. Trigger page-specific refresh on page open.
3. On save, use a write method with verification where correctness matters.
4. Handle `AdminWriteResult` for user-facing success/failure messaging.

=== Channels Page

Recommended open flow:

[source,java]
----
// 1) Snapshot render (instant)
renderChannels(admin.getChannelSnapshots());

// 2) Refresh slots on page open
admin.refreshChannels().thenAccept(channels -> {
    renderChannels(channels);
});
----

Recommended writes:

[source,java]
----
// rename slot 2
admin.setChannelName(2, "meshMKT")
     .thenAccept(applied -> refreshChannelsUi());

// set password/psk on slot 2
admin.setChannelPassword(2, "shared-secret")
     .thenAccept(applied -> refreshChannelsUi());

// full channel payload update with structured status
admin.setChannelResult(2, updatedChannel, true)
     .thenAccept(this::handleWriteResult);
----

=== Security Page

Recommended open flow:

[source,java]
----
admin.getConfigSnapshot(ConfigType.SECURITY_CONFIG)
     .ifPresent(this::renderSecuritySnapshot);

admin.refreshSecurityConfig().thenAccept(config -> {
    if (config.hasSecurity()) {
        renderSecurity(config.getSecurity());
    }
});
----

Recommended write:

[source,java]
----
Config.SecurityConfig security = Config.SecurityConfig.newBuilder()
        .setAdminChannelEnabled(true)
        .build();

admin.setSecurityConfig(security, true)
     .thenAccept(applied -> {
         if (applied) showSuccess("Security settings applied.");
         else showWarning("Security settings accepted but not verified.");
     });
----

=== MQTT Page

MQTT in Meshtastic is a module config section, not a top-level config section.

Recommended open flow:

[source,java]
----
admin.getModuleConfigSnapshot(ModuleConfigType.MQTT_CONFIG)
     .ifPresent(snapshot -> {
         if (snapshot.hasMqtt()) {
             renderMqtt(snapshot.getMqtt());
         }
     });

admin.refreshMqttConfig().thenAccept(moduleConfig -> {
    if (moduleConfig.hasMqtt()) {
        renderMqtt(moduleConfig.getMqtt());
    }
});
----

Recommended write:

[source,java]
----
ModuleConfig.MQTTConfig mqtt = ModuleConfig.MQTTConfig.newBuilder()
        .setEnabled(true)
        .setAddress("mqtt.example.com")
        .setUsername("mesh-user")
        // set password/tls/root fields as required by your deployment
        .build();

admin.setMqttConfig(mqtt, true).thenAccept(applied -> {
    if (applied) {
        showSuccess("MQTT settings applied.");
    } else {
        showWarning("MQTT settings accepted but read-back mismatch.");
    }
});
----

How MQTT settings are used in a client app:

- Your app writes MQTT module settings via `AdminService`.
- The radio firmware applies those settings and controls MQTT bridge behavior.
- Your Java app usually does not need a separate MQTT client just to enable radio MQTT forwarding.
- If your app also consumes broker traffic directly (analytics/backend tooling), run a separate MQTT client in your app process; that is independent of radio admin configuration.

=== Write Result Handling Pattern

Reusable UI handler:

[source,java]
----
private void handleWriteResult(AdminWriteResult result) {
    switch (result.status()) {
        case VERIFIED_APPLIED -> showSuccess(result.operation() + ": verified.");
        case ACCEPTED -> showSuccess(result.operation() + ": accepted.");
        case REJECTED -> showError(result.operation() + ": rejected (" + result.message() + ")");
        case TIMEOUT -> showError(result.operation() + ": timeout (" + result.message() + ")");
        case VERIFICATION_FAILED -> showWarning(result.operation() + ": verification failed.");
        case FAILED -> showError(result.operation() + ": failed (" + result.message() + ")");
    }
}

== Request Lifecycle Events

`MeshtasticEventListener` exposes request lifecycle updates through:

[source,java]
----
client.addEventListener(new MeshtasticEventListener() {
    @Override
    public void onRequestLifecycleUpdate(RequestLifecycleEvent event) {
        switch (event.getStage()) {
            case SENT -> uiLog("Sent " + event.getPort() + " -> " + MeshUtils.formatId(event.getDestinationNodeId()));
            case ACCEPTED -> uiLog("Accepted request #" + event.getRequestId());
            case REJECTED -> showError("Rejected: " + event.getMessage());
            case TIMED_OUT -> showError("Timed out: " + event.getMessage());
            case PAYLOAD_RECEIVED -> uiLog("Payload observed for request #" + event.getRequestId());
            case CANCELLED -> uiLog("Cancelled request #" + event.getRequestId());
            case FAILED -> showError("Request failed: " + event.getMessage());
        }
    }
});
----

This gives one consistent contract for request outcomes across admin reads/writes and utility request methods.

== Admin Model Update Events

For settings UIs, `MeshtasticEventListener` now exposes `onAdminModelUpdate(...)` for live model changes.

When this event fires:

- `ADMIN_APP` responses update owner/config/channel/module/metadata
- local `FromRadio` snapshots update local node/config/channel/module/metadata

[source,java]
----
client.addEventListener(new MeshtasticEventListener() {
    @Override
    public void onAdminModelUpdate(AdminModelUpdateEvent event) {
        switch (event.getSection()) {
            case CHANNEL -> {
                Integer slot = event.getChannelIndex();
                if (slot != null) {
                    admin.refreshChannel(slot).thenAccept(ch -> renderChannelRow(slot, ch));
                } else {
                    admin.refreshChannels().thenAccept(MainPresenter.this::renderChannels);
                }
            }
            case OWNER -> admin.refreshOwner().thenAccept(MainPresenter.this::renderOwner);
            case CONFIG -> admin.refreshCoreConfigs().thenAccept(cfg -> renderConfig(admin.getConfigSnapshots()));
            case MODULE_CONFIG -> admin.refreshModules().thenAccept(m -> renderModules(admin.getModuleConfigSnapshots()));
            case DEVICE_METADATA -> admin.refreshMetadata().thenAccept(MainPresenter.this::renderMetadata);
            case LOCAL_NODE -> { /* optional local node update handling */ }
        }
    }
});
----

[TIP]
====
Firmware does not always push channel/config changes made directly on-device. For deterministic settings pages:

1. refresh on page-open,
2. optionally poll while page is visible,
3. stop polling when page closes.
====

== OTA Framework (Serial-First)

[IMPORTANT]
====
OTA support in this Java client is currently **advanced/experimental**.

For most production users, the recommended firmware update path is the official Meshtastic web flasher/updater.
Use the OTA APIs here when you explicitly need programmatic control and can validate behavior on your target
hardware/firmware combinations.
====

The OTA API is split into two parts:

- `AdminService` requests OTA mode (`requestOtaModeResult(...)`).
- `OtaService` orchestrates hash + OTA mode request + uploader strategy.

This keeps OTA transport-agnostic. You can use a Linux serial uploader now and reuse the same orchestration later for BLE.
It also means the repository does not currently ship one universally ready OTA uploader for every board/transport;
that last-mile implementation is expected to be provided and validated per environment.

=== OTA Orchestration Example

[source,java]
----
OtaService ota = client.getOtaService();

OtaRequest request = OtaRequest.builder()
        .targetNodeId(targetNodeId)
        .firmwarePath(Path.of("/path/to/firmware.bin"))
        .otaMode(OTAMode.OTA_BLE)
        .rebootWait(Duration.ofSeconds(5))
        .build();

OtaUploadStrategy uploader = new CommandLineOtaUploadStrategy(List.of(
        "esptool.py",
        "--port", "/dev/ttyUSB0",
        "write_flash",
        "0x10000",
        "{firmware}"
));

OtaSession session = ota.start(
        request,
        uploader,
        progress -> {
            switch (progress.stage()) {
                case HASHING_FIRMWARE -> showStatus("Hashing firmware...");
                case REQUESTING_OTA_MODE -> showStatus("Requesting OTA mode...");
                case WAITING_FOR_REBOOT -> showStatus("Waiting for reboot...");
                case UPLOADING_FIRMWARE -> showProgress(progress.bytesSent(), progress.totalBytes());
                case COMPLETED -> showStatus("OTA complete");
                case FAILED -> showError(progress.message());
                case CANCELLED -> showWarning("OTA cancelled");
                default -> { }
            }
        }
);

session.resultFuture().thenAccept(result -> {
    if (result.isSuccess()) {
        showSuccess("OTA finished");
    } else {
        showError("OTA failed: " + result.message());
    }
});
----

=== OTA Design Notes

- OTA upload implementation is intentionally externalized via `OtaUploadStrategy`.
- Linux serial release can ship with `CommandLineOtaUploadStrategy` + CLI tooling.
- BLE transport can later provide another `OtaUploadStrategy` without changing OTA API or UI flow.
- This separation is intentional so contributors can add hardware-specific or platform-specific OTA implementations
  without forcing one uploader stack into core.

=== OTA Support Matrix (Current)

[cols="1,1,2",options="header"]
|===
| Path
| Status
| Notes

| `CommandLineOtaUploadStrategy`
| Usable
| Requires external uploader tool installed/configured by app/developer (for example board-specific CLI flash utility).

| `SerialXmodemUploadStrategy`
| Experimental
| In-process protocol loop is implemented, but requires a concrete `XmodemDuplex` adapter bound to real radio OTA link behavior.

| BLE OTA uploader strategy
| Planned
| Intended future strategy once BLE transport path is implemented/stabilized.
|===

Recommended product behavior:

- If OTA path/model is not explicitly supported/tested, direct users to the official web flasher.
- Keep OTA actions behind an “Advanced” UI section until path support is validated.

=== In-Process XMODEM Strategy (Experimental)

For environments where you do not want external command-line tools, use the experimental in-process uploader:

[source,java]
----
OtaService ota = client.getOtaService();

// Firmware path is defined here and passed into the upload strategy via OtaUploadContext.
OtaRequest request = OtaRequest.builder()
        .targetNodeId(targetNodeId)
        .firmwarePath(Path.of("/path/to/firmware.bin"))
        .otaMode(OTAMode.OTA_BLE)
        .rebootWait(Duration.ofSeconds(5))
        .build();

MeshtasticXmodemDuplex duplex = client.createXmodemDuplex();
duplex.reset(); // clear stale control events before a new upload attempt

OtaUploadStrategy uploader = new SerialXmodemUploadStrategy(
        duplex,
        Duration.ofSeconds(2),
        10
);

OtaSession session = ota.start(request, uploader, progress -> {
    // update UI / logs
});
----

Notes:

- This strategy is CLI-free and uses top-level protobuf envelopes: `ToRadio.xmodemPacket` and `FromRadio.xmodemPacket`.
- `OtaRequest.firmwarePath` is the source of the upload file for all strategies.
- `duplex.reset()` is recommended at session start to avoid consuming stale controls from prior attempts.
- Keep this path experimental until validated against your target hardware/firmware variants.

== BLE Transport SPI (Skeleton)

The core includes a BLE transport skeleton with a pluggable backend SPI:

- `BleConfig` (connection IDs/UUIDs/timeouts/reconnect policy)
- `BleLinkBackend` (backend adapter contract to your BLE library)
- `BleTransport` (meshtastic framing + reconnect + lifecycle wiring)

Example bootstrap:

[source,java]
----
BleConfig cfg = BleConfig.builder()
        .deviceId("AA:BB:CC:DD:EE:FF")
        .serviceUuid("service-uuid")
        .toRadioCharacteristicUuid("to-radio-char-uuid")
        .fromRadioCharacteristicUuid("from-radio-char-uuid")
        .connectTimeout(Duration.ofSeconds(10))
        .reconnectBackoff(Duration.ofSeconds(2))
        .autoReconnect(true)
        .build();

BleLinkBackend backend = new MyBleBackend(); // your adapter for a concrete BLE stack
client.connect(new BleTransport(cfg, backend));
----

This keeps the core API stable while allowing platform-specific BLE implementations without hard-coding one dependency.

[IMPORTANT]
====
BLE support is not 100% implemented out of the box. `BleTransport` provides the Meshtastic-facing transport behavior,
but a real platform BLE driver/backend still needs to be supplied by the application or by a community module.
====

=== Candidate BLE Backends

Depending on OS/runtime, a contributor or application can wire `BleLinkBackend` to a concrete Java BLE library such as:

- Gluon Attach BluetoothLE for Gluon/JavaFX-oriented applications
- TinyB for Linux/BlueZ environments
- a BlueZ D-Bus integration built on a Java D-Bus library for Linux-specific apps

[NOTE]
====
These are examples of possible backend choices, not official bundled dependencies of this project. The right backend
depends on packaging constraints, runtime environment, and the target operating system.
====
